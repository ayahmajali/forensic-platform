Introduction

1.1 Background

Digital forensics is the scientific discipline of recovering, preserving, and analysing electronically stored information (ESI) in a way that is legally admissible. As cybercrime, data breaches, and insider threats proliferate, the demand for automated, reproducible, and scalable forensic tooling continues to grow in both law‑enforcement and enterprise contexts.

Traditional forensic suites such as Autopsy, FTK (Forensic Toolkit), and EnCase are powerful but require installation on a forensic workstation, carry significant hardware requirements, and operate exclusively through desktop GUIs. A web‑based platform eliminates these barriers: any authorised investigator with a browser can submit evidence and receive a structured, shareable report without local tool installation.

1.2 Motivation

🖥️

Desktop Tool Limitations

Enterprise tools like Autopsy require Windows/macOS installation and are impractical for cloud or remote environments.

⚡

Automation Gap

Investigators still manually run multiple CLI tools (mmls, fls, exiftool). A unified pipeline reduces human error.

🤖

AI-Augmented Triage

LLM models can now produce human‑readable forensic summaries from structured findings, accelerating triage time.

📱

Mobile Access

Field investigators need responsive interfaces accessible on tablets and phones without VPN or desktop apps.

1.3 Objectives

  1. Build a web-based forensic investigation platform accessible via any modern browser.
  2. Integrate The Sleuth Kit CLI tools into a fully automated analysis pipeline.
  3. Support disk images (E01, DD/RAW, IMG, ISO) and logical files (PDF, DOCX, JPG, etc.).
  4. Extract browser history, EXIF metadata, multimedia artefacts, and deleted files automatically.
  5. Generate AI-powered investigation summaries using the OpenAI GPT-4o-mini model.
  6. Produce self-contained, interactive HTML reports with embedded timeline and search.
  7. Provide a RESTful API for integration with SIEM / SOAR platforms.
  8. Deploy on Render.com with full CI/CD via GitHub and maintain chain-of-custody hashing.

2.3 Data Flow

The investigation lifecycle follows eight sequential stages, all managed by run_full_analysis() as a FastAPI BackgroundTask:

1

Evidence Upload & Job Creation

Client submits multipart/form-data POST to /api/investigate. File is saved to uploads/ with a UUID job ID prefix. A job record is inserted into the in-memory store with status queued.

2

Evidence Type Detection

ForensicAnalyzer inspects file extension AND reads the first 512 bytes (magic bytes) to classify the evidence as disk_image, e01, iso, or logical_file.

hashlibmagic bytes
3

Cryptographic Hashing

MD5, SHA-1, and SHA-256 digests computed in 64 KB chunks for large-file compatibility. Hashes establish evidence integrity and enable chain-of-custody verification.

hashlib (Python stdlib)
4

Disk Image Analysis (conditional)

If the evidence is a disk image, DiskAnalyzer executes mmls (partition table), fsstat (filesystem info), fls -r (all files), fls -rd (deleted files), ils (inodes), and tsk_recover (file extraction).

mmlsfsstatflsilstsk_recover
5

Artifact Extraction

ArtifactExtractor walks the recovered directory, reads SQLite browser databases (Chrome History, Firefox places.sqlite), extracts EXIF using ExifTool, and catalogues all multimedia and document files.

sqlite3exiftoolphotorecforemost
6

Timeline Construction

TimelineBuilder generates a MAC-time (Modified/Accessed/Changed) timeline using fls -r -m body file + mactime. Falls back to os.stat() filesystem timestamps if TSK is unavailable.

fls -r -mmactimeos.stat
7

Keyword Search

KeywordSearchEngine runs five parallel search strategies per keyword: filename scan, text-file content, browser history, EXIF metadata, and SQLite cell scan. Supports email and URL pattern extraction via regex.

regexsqlite3os.walk
8

AI Summary + Report Generation

AISummarizer calls OpenAI GPT-4o-mini with a structured forensic prompt. ReportGenerator assembles all findings into a self-contained HTML report (embedded base64 images, interactive JS search, responsive sidebar).

OpenAI APIHTML/CSS/JSbase64

Technology Stack & Justification

Every technology was selected based on four criteria: forensic correctness, performance, deployment simplicity, and community adoption in the digital forensics field.

3.1 Backend Framework — FastAPI (Python 3.11)

Python 3.11 FastAPI 0.109+ Uvicorn ASGI Async / Await Pydantic

FastAPI is a modern, high-performance web framework for Python based on standard type hints. It was chosen over alternatives (Flask, Django, Express) for the following reasons:

Feature FastAPI Flask Django Express (Node)
Async I/O support ✔ Native ✗ Extension ✗ Limited ✔ Native
Automatic OpenAPI docs ✔ Built-in ✗ Plugin ✗ Plugin ✗ Manual
Type validation (Pydantic) ✔ Built-in ✗ ✗ ✗
Background task support ✔ Native ✗ Celery req. ✗ Celery req. ✗ Bull req.
Python ecosystem (subprocess, hashlib) ✔ Full ✔ Full ✔ Full ✗ N/A
Performance (requests/sec) ~58,000 ~16,000 ~12,000 ~42,000
ℹ️
Why not Node.js / Express? The entire forensic toolchain (Sleuth Kit, ExifTool, PhotoRec) is invoked via Python's subprocess module. Python's hashlib, sqlite3, and pathlib standard library modules are ideal for forensic file analysis. Switching to Node.js would introduce unnecessary complexity.

Key FastAPI Features Used

  • BackgroundTasks — runs the 8-step analysis pipeline asynchronously without blocking the HTTP response, allowing the UI to poll for progress via /api/status/{job_id}.
  • UploadFile + aiofiles — async chunked file writes prevent the server from blocking during large evidence uploads (multi-GB disk images).
  • CORSMiddleware — allows the frontend (or future mobile app) to call the API from any origin.
  • StaticFiles mount — serves the /static, /reports, and /recovered directories efficiently without custom route handlers.
  • Jinja2Templates — renders index.html server-side, enabling dynamic injection of environment-specific configuration.
# Example: Non-blocking file upload + background analysis
@app.post("/api/investigate")
async def start_investigation(
    background_tasks: BackgroundTasks,
    evidence: UploadFile = File(...),
    keywords: str = Form(default="password,bitcoin,secret")
):
    job_id = str(uuid.uuid4())[:8]
    async with aiofiles.open(evidence_path, 'wb') as f:
        await f.write(await evidence.read())   # async I/O
    background_tasks.add_task(              # non-blocking
        run_full_analysis, job_id, evidence_path, ...
    )
    return JSONResponse({"job_id": job_id, "status": "queued"})

3.2 The Sleuth Kit (TSK)

mmls fsstat fls ils tsk_recover mactime Open Source (GPL v2)

The Sleuth Kit (TSK), originally created by Brian Carrier and maintained by the forensic community, is the industry-standard open-source framework for file system forensics. It supports FAT12/16/32, exFAT, NTFS, ext2/3/4, HFS+, APFS, ISO 9660, and more.

ToolFunctionPlatform Module
mmls Reads partition table (MBR/GPT). Returns slot, start sector, end sector, length, and partition type for each entry. disk_analysis.py → run_mmls()
fsstat Extracts filesystem metadata: type (NTFS, FAT32, ext4), volume name, block size, last mount time. disk_analysis.py → run_fsstat()
fls Lists files and directories from a filesystem image. -r for recursive, -d for deleted-only, -m for body file generation. disk_analysis.py → run_fls()
ils Lists allocated and unallocated inode entries. Useful for detecting anti-forensics (inode gaps). disk_analysis.py → run_ils()
tsk_recover Recovers all allocated (and optionally deleted) files from a disk image to an output directory. disk_analysis.py → recover_files()
mactime Converts fls -m body files into sorted timeline entries (MAC — Modified, Accessed, Changed). timeline_builder.py → build_tsk_timeline()
✅
Why TSK over Python-only libraries? Python libraries such as pytsk3 (Python bindings for TSK) or dfvfs (Digital Forensics Virtual File System) exist, but invoke the same underlying TSK C libraries. Using the native CLI tools directly avoids binding version conflicts, supports the widest range of filesystem types, and produces output identical to what court-accepted tools generate.

Resilient Subprocess Execution

All TSK commands are wrapped in DiskAnalyzer._run_cmd() with:

  • Configurable timeouts (120–300 seconds depending on operation).
  • Graceful fallback when tools are not installed (FileNotFoundError caught).
  • UTF-8 decoding with errors="replace" to handle binary output safely.
  • Return code inspection to distinguish tool errors from empty results.

3.4 AI Summarization — OpenAI GPT-4o-mini

OpenAI API GPT-4o-mini Temperature 0.3 Max tokens 1000

The platform integrates OpenAI's GPT-4o-mini model to generate human-readable forensic investigation summaries. The model was chosen over GPT-4o (full) because:

  • Cost efficiency — 15× cheaper per token while maintaining strong reasoning for structured forensic prompts.
  • Latency — typically responds in 3–8 seconds vs 15–30 seconds for GPT-4o.
  • Instruction following — reliably produces structured Markdown output (Executive Summary, Key Findings, Recommendations) with temperature=0.3.

Prompt Engineering Strategy

The system prompt declares the model as a "senior digital forensics investigator". The user prompt provides structured JSON-like context including evidence hashes, file counts, browser domains, GPS count, and suspicious keywords, then requests a specific 5-section Markdown report. This constrained output approach ensures consistent, court-ready formatting.

⚠️
Graceful Fallback: If OPENAI_API_KEY is not configured or the API call fails, the platform automatically generates a structured template-based summary using AISummarizer._template_summary(). This ensures the investigation pipeline always completes successfully even without network connectivity.

Why Vanilla JS over React/Svelte/SolidJS? The original brief mentioned React, Svelte, or SolidJS. 4.4 TimelineBuilder (modules/timeline_builder.py)

Generates a MAC-time timeline (Modified, Accessed, Changed) — the cornerstone of any temporal forensic investigation.

Timeline Construction Logic
Method 1 — TSK pipeline (preferred):
fls -r -m / [image]  →  timeline.body  →  mactime -b timeline.body -d
   ↓ Parse CSV output (date, activity, filename, size, inode)

Method 2 — Body file direct parse (mactime unavailable):
Parses fls -m body file format:
  MD5|name|inode|mode|UID|GID|size|atime|mtime|ctime|crtime
  Converts Unix timestamps to UTC strings

Method 3 — Filesystem fallback (TSK unavailable):
os.walk(recovered_path)  →  os.stat(file)
  → st_mtime / st_atime / st_ctime timestamps

Smart Keyword Routing

if "@" in kw and "." in kw:
    hits = search_emails(kw)           # email pattern
elif kw.startswith(".") and len(kw) <= 6:

Analysis Pipeline — Technical Detail

The run_full_analysis() coroutine in main.py is the orchestrator of all eight pipeline stages. It is launched as a FastAPI BackgroundTask immediately after the evidence upload is acknowledged, preventing HTTP timeout on large files.

StepFunctionProgress %Failure Mode
1. Upload & Job InitFastAPI route handler0%HTTP 422 if no file provided
2. Evidence DetectionForensicAnalyzer.detect_evidence_type()10%Falls back to logical_file
3. Hash ComputationForensicAnalyzer.compute_hashes()15%I/O error → job fails
4. Disk AnalysisDiskAnalyzer.run_full_disk_analysis()30%TSK missing → empty results, continues
5. File RecoveryDiskAnalyzer.recover_files()45%tsk_recover missing → logical copy
6. Artifact ExtractionArtifactExtractor.extract_all()60%SQLite errors caught per-file
7. TimelineTimelineBuilder.build_timeline()70%Falls back to filesystem timeline
8. Keyword SearchKeywordSearchEngine.search_all()80%Empty results, continues
9. AI SummaryAISummarizer.generate_summary()88%Falls back to template summary
10. Report GenerationReportGenerator.generate()100%I/O error → job fails
✅
Resilience Design: Every stage is wrapped in try/except. If a non-critical stage fails (TSK unavailable, ExifTool missing, OpenAI unreachable), the pipeline logs a warning and continues. Only I/O errors on the evidence file or report directory cause the job to transition to failed status. This ensures the platform is usable even in environments where forensic tools are partially installed.

Security & Forensic Integrity

7.1 Evidence Integrity

Forensic investigations must demonstrate that evidence has not been altered between acquisition and analysis. The platform implements the following integrity measures:

  • Triple hashing — MD5, SHA-1, and SHA-256 computed immediately upon upload before any analysis begins. These values are embedded in every report and returned by /api/status.
  • Read-only analysis — TSK tools operate on the original evidence file without modification. Only tsk_recover writes to a separate output directory.
  • Immutable upload path — uploaded files are saved with a UUID prefix and never overwritten; deletion is only possible via the explicit DELETE API.

7.2 API Key Security

  • OpenAI API key is stored only in os.environ (process memory); never written to disk.
  • MongoDB URI and other secrets are injected via platform environment variables (Render.com dashboard), never committed to the git repository.
  • The .env file is listed in .gitignore; only .env.example is committed.

7.3 CORS & Input Validation

  • CORS — allow_origins=["*"] is used for the current prototype. Production deployment should restrict origins to known investigator domains.
  • File sanitisation — evidence.filename.replace(" ", "_") prevents path traversal via filename injection.
  • Job isolation — each job gets a unique UUID directory under uploads/, recovered/, and reports/; cross-job file access is structurally prevented.

7.4 Production Hardening (Recommended)

⚠️
The following hardening steps are recommended before using this platform in a live investigative environment:
  • Add JWT/OAuth2 authentication middleware to all API routes.
  • Implement rate limiting (e.g., slowapi) to prevent abuse.
  • Switch in-memory job store to Redis or MongoDB for persistence and multi-instance support.
  • Enable HTTPS-only (enforced by Render.com and most cloud hosts).
  • Log all investigation activities to an append-only audit log with timestamps.
  • Restrict CORS to investigator dashboard domain only.

Conclusion

This graduate project demonstrates that professional-grade digital forensic investigation capabilities can be delivered as a web application, making forensic analysis accessible to investigators without specialist hardware or local tool installation.

The platform successfully integrates seven distinct technology domains: web framework engineering (FastAPI), disk forensics (The Sleuth Kit), metadata analysis (ExifTool), browser artifact parsing (SQLite3), temporal analysis (MAC-time timeline), multi-strategy keyword search, and AI-augmented reporting (OpenAI GPT-4o-mini).

All eight pipeline stages operate with graceful degradation — the system produces useful results even when individual tools are unavailable. The mobile-responsive interface, comprehensive REST API, and self-contained HTML reports make the platform suitable for both field investigators and integration with SIEM/SOAR automation platforms.

🎓
Key Academic Contributions:
  1. A complete, working forensic platform designed and implemented from scratch.
  2. Novel integration of LLM (GPT-4o-mini) into the forensic analysis pipeline for automated triage summarisation.
  3. Demonstration of how traditional CLI forensic tools (TSK, ExifTool) can be orchestrated by a modern async Python web framework.
  4. A fully mobile-responsive forensic interface — a gap not addressed by existing open-source tools.
  5. Complete REST API enabling forensic automation and SOAR integration.
