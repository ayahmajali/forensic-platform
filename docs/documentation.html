Introduction

1.1 What Is This Platform?

The Digital Forensics Investigation Platform is a full-stack web application that automates the end-to-end process of digital evidence analysis. An investigator uploads a piece of digital evidence — a raw disk image, an EnCase E01 file, an ISO, or a logical file — and the platform automatically:

  • Detects the evidence type using magic bytes and file signatures
  • Computes cryptographic hashes (MD5, SHA-1, SHA-256) for integrity verification
  • Runs full disk analysis: partition table, filesystem info, all files, deleted files, inodes
  • Recovers deleted files using tsk_recover
  • Extracts browser history from Chrome, Firefox, and Edge SQLite databases
  • Pulls EXIF metadata (camera model, GPS, timestamps) from image files
  • Catalogs multimedia (images, videos) and documents (PDF, Word, Excel)
  • Builds a forensic MAC-time timeline using fls and mactime
  • Performs deep keyword search across filenames, content, browser history, metadata, and SQLite databases
  • Optionally generates an AI-powered investigation summary via OpenAI GPT
  • Produces a rich, interactive HTML forensic report

1.2 Target Users

This platform is designed for digital forensics students, academic researchers, and investigators who need a self-hosted, open-source alternative to commercial tools like Autopsy, FTK, and EnCase — accessible through a modern browser without any local installation.

1.3 Key Differentiators

🌐 Browser-Based
No installation needed on the analyst's machine. Upload evidence from any device.
🤖 AI-Powered Summaries
GPT-4o-mini generates professional narrative forensic reports from raw findings.
🔍 Multi-Source Search
Keyword search spans filenames, content, browser URLs, metadata, and SQLite DBs.
📊 Interactive Reports
HTML reports with lightboxes, video players, filterable tables, and timelines.
🏗️ Modular Architecture
Each forensic capability is a separate Python class, easy to extend or replace.
☁️ Cloud-Ready
Deploys to Render.com backend + Cloudflare Pages frontend out of the box.

2.3 Request Data Flow

1

File Upload

Browser POSTs multipart form to /api/upload-evidence. FastAPI saves to storage/uploads/{uuid}/ and creates an evidence document in MongoDB with status uploaded.

2

Analysis Trigger

Client POSTs to /api/analyze/{id}. FastAPI launches the full pipeline as a BackgroundTask and immediately returns 202 Accepted.

3

Forensic Pipeline Execution

Background task runs all modules sequentially: type detection → hashing → disk analysis → file recovery → artifact extraction → timeline → keyword search → AI summary → report generation. Progress is written to MongoDB in real time.

4

Status Polling

Frontend polls /api/results/{id} every 3 s. Response includes progress (0–100) and log[] entries for live display.

5

Results & Report

On completion, the frontend navigates to the interactive HTML report at /api/report/{id} or fetches structured JSON from /api/results/{id} for the dashboard view.

Technology Stack & Justification

Every technology was deliberately chosen. This section documents each choice, explains its features used in the project, and justifies why it was selected over alternatives.

⚡

FastAPI — Backend Web Framework

Python 3.11 · FastAPI ≥ 0.109 · Uvicorn
Why FastAPI? Forensic analysis involves long-running CPU-bound tool calls (Sleuth Kit, ExifTool) and I/O-bound database writes. FastAPI's async/await model lets the server handle hundreds of status-polling requests concurrently while a background worker processes the disk image — impossible with synchronous Flask or Django.

Features Used in This Project

🔄 Async Background Tasks
BackgroundTasks runs the full forensic pipeline without blocking the HTTP response. Analysis can take minutes for large images.
📋 Pydantic Data Models
All request/response bodies are validated by Pydantic models (EvidenceCreate, AnalysisResult, SearchHit). Ensures type safety throughout.
📁 Multipart File Upload
python-multipart handles streaming uploads up to 4 GB, saving evidence files to disk with UUID-named directories.
🔒 CORS Middleware
CORSMiddleware allows the Cloudflare Pages frontend (different domain) to call the Render.com API securely.
📦 Static File Mounting
StaticFiles serves the /static, /storage/reports, and /storage/recovered directories so generated reports and images are downloadable.
🧩 Router Separation
Each domain (evidence, analysis, search, report, health) lives in its own APIRouter, keeping main.py clean and each route file focused.
♻️ Lifespan Context Manager
The modern @asynccontextmanager lifespan pattern replaces deprecated on_event hooks for clean MongoDB connect/disconnect at startup and shutdown.
📖 Auto Documentation
Swagger UI at /docs and ReDoc at /redoc are auto-generated from code, giving the team live API docs with no extra effort.

Why Not Flask / Django?

CriterionFastAPIFlaskDjango

🍃

Motor & MongoDB — Async Database Layer

Motor (async) · PyMongo ≥ 4.6 · MongoDB Atlas
Why MongoDB? Forensic analysis results are deeply nested, variable-length JSON documents. A single analysis result may contain 10,000 file entries, 500 browser history records, and a multi-page AI summary — all with different shapes. MongoDB's document model stores this naturally without joins, while Motor's async driver integrates perfectly with FastAPI's event loop.

Features Used in This Project

📄 Document Storage
The analysis_results collection stores the complete result of a forensic job as a single BSON document — no complex JOIN queries needed to retrieve everything.
🔍 Full-Text Search Index
The search_index collection uses a compound text index on content and file_path fields, enabling fast $text queries across all indexed evidence files.
⚡ Async Client (Motor)
Motor provides a fully async PyMongo wrapper. All DB operations use await, preventing the event loop from blocking during slow Atlas queries.
🏊 Connection Pooling
The Database singleton class configures maxPoolSize=20, reusing TCP connections across requests for efficiency.
📊 Performance Indexes
Created at startup: upload_date and status indexes on evidence; unique evidence_id index on analysis_results.
🔗 Flexible Schema
Different evidence types produce different result shapes. MongoDB allows some jobs to have partition tables while others (logical files) skip that field entirely.

Three MongoDB Collections

CollectionPurposeKey Fields
evidenceEvidence file registryid, filename, type, status, hash_md5, hash_sha256, upload_date
analysis_resultsFull forensic resultsevidence_id, partitions, files_list, deleted_files, browser_history, metadata, timeline, ai_summary, report_url

🗂️

The Sleuth Kit (TSK) — Disk Image Analysis

mmls · fsstat · fls · ils · tsk_recover · mactime
Why Sleuth Kit? TSK is the gold-standard, open-source forensic toolkit trusted in real investigations and accepted by courts. It reads raw disk images (E01, DD, ISO) at the block level without mounting them — essential for forensic soundness because mounting can alter timestamps and metadata.

TSK Commands Used

CommandWhat It DoesOutput Used For
mmlsReads partition table (MBR/GPT). Lists all partitions with start/end sectors, lengths, and descriptions.Partition listing, byte-offset calculation for other tools
fsstatDisplays filesystem metadata: type (NTFS/FAT32/ext4), volume name, block size, block count, last mount time.Filesystem info section of the report
fls -rRecursively lists all files and directories in the filesystem, including their inodes.Complete file listing (capped at 5,000)
fls -r -dLists only deleted files (unallocated inode entries). The r/- flag marks deleted entries.Deleted files section, highlighted in red
fls -r -m /Outputs body file format: MD5|name|inode|mode|UID|GID|size|atime|mtime|ctime|crtime — for timeline construction.Input to mactime
ilsLists inode information directly from disk — detects inodes allocated without directory entries (orphaned files).Inode listing (capped at 1,000)
tsk_recoverRecovers files from a disk image into an output directory. Handles both active and deleted/unallocated files.Populates storage/recovered/{job_id}/
mactimeConverts body file into a chronological MAC-time timeline (Modified / Accessed / Changed). Output is CSV.Timeline section of the report (capped at 2,000 events)
ℹ️ All TSK tools are invoked via Python's subprocess.run() with explicit timeouts (120–300 s), capture_output=True, and encoding="utf-8", errors="replace" to handle non-UTF filenames gracefully. If a tool is not installed, the platform falls back to filesystem-based analysis.

Supported Evidence Formats

E01 / EWF
EnCase Evidence Format. Magic bytes: EVF. Most common in professional forensics.
DD / RAW / IMG
Sector-by-sector raw disk image. Detected by MBR signature 0x55AA at byte 510.
ISO 9660
Optical disc image. Magic: CD001 at offset 1. Used for CD/DVD evidence.
Logical Files
Individual files (PDF, DOCX, JPG, etc.) — no disk analysis, only artifact extraction applies.
📷

ExifTool — Metadata Extraction

Phil Harvey's ExifTool · CLI via subprocess
Why ExifTool? ExifTool reads metadata from over 100 file formats and extracts forensically valuable fields: GPS coordinates, camera make/model/serial number, software used, creation and modification timestamps. It outputs clean JSON, making programmatic parsing trivial. No Python-native alternative covers as many formats.

Fields Extracted

FieldForensic Significance
Make, ModelIdentifies the device that created the image — useful for linking photos to a specific phone or camera
SerialNumberUniquely identifies a specific camera unit
DateTimeOriginalWhen the image was captured — embeds in the image regardless of filesystem timestamps
CreateDate, ModifyDateFile creation and last modification in the image's EXIF stream
GPSLatitude, GPSLongitudeGeographic location where the photo was taken — powerful investigative intelligence
GPSPositionCombined GPS string for display
SoftwareEditing software used (e.g., Adobe Photoshop) — indicates image manipulation

ExifTool is called with -json flag per file, then json.loads() parses the result. Falls back to os.stat() timestamps if ExifTool is unavailable.

🗄️

SQLite3 — Browser Artifact Parsing & Search

Python stdlib sqlite3 · No external dependency
Why SQLite3? Modern browsers (Chrome, Firefox, Edge) store their history, cookies, and bookmarks in SQLite databases. Python's built-in sqlite3 module lets the platform read these databases directly from recovered files without any external tools. It is also used by the keyword search engine to scan any SQLite database found in recovered files.

Browser Database Schemas Parsed

BrowserDatabase FileTables QueriedFields Extracted
Chrome / EdgeHistoryurls JOIN visitsurl, title, visit_count, visit_time (microseconds since 1601-01-01)
Firefoxplaces.sqlitemoz_places JOIN moz_historyvisitsurl, title, visit_count, visit_date (microseconds since Unix epoch)

Timestamp Conversion

# Chrome: microseconds since Windows epoch (1601-01-01)
ts = (row["visit_time"] / 1_000_000) - 11_644_473_600
datetime.utcfromtimestamp(ts)

# Firefox: microseconds since Unix epoch (1970-01-01)
ts = row["visit_date"] / 1_000_000
datetime.utcfromtimestamp(ts)

SQLite Magic Byte Detection

Before attempting to open a file as SQLite, the engine checks for the 16-byte magic string SQLite format 3 to avoid errors on non-database files with misleading names.

Deep SQLite Search

The keyword search engine also scans any SQLite file found in recovered data — not just browser databases. It reads all tables up to 1,000 rows each and searches every string column for the keyword, covering app databases, messaging apps, and system logs.

🤖

OpenAI GPT-4o-mini — AI Summarization

gpt-4o-mini (default) · OpenAI Chat Completions API
Why GPT-4o-mini? Forensic data is vast and technical. Presenting an investigator with 10,000 file entries requires human-readable interpretation. GPT-4o-mini converts structured findings into a professional 5-section narrative report in seconds. It is selected over GPT-4 for cost efficiency (much cheaper per token) while still producing high-quality professional text for this summarization task.

How the AI Summary Works

📝 Structured Prompt
The prompt is constructed from real analysis data: file counts, deleted files, browser URLs, GPS count, camera models, and keyword hits. No hallucination risk — all facts come from the actual scan.
🔌 Direct HTTPS API Call
Uses Python's urllib.request (stdlib) — not the openai package — to POST to https://api.openai.com/v1/chat/completions. Reduces dependency weight.
🛡️ Graceful Fallback
If OPENAI_API_KEY is not set, or the API call fails, _template_summary() generates an equally structured Markdown report using only local data — no AI required.
📄 5-Section Output
The AI is instructed to produce: Executive Summary, Key Findings, Suspicious Activity, Recommended Follow-up, and Evidence Integrity Statement — professional court-ready language.

Prompt Engineering Strategy

System: "You are a professional digital forensics investigator writing official case reports."

User prompt includes:
  - Evidence filename, type, SHA256, MD5, size
  - Count of recovered files, images, videos, documents
  - Count of deleted files and timeline events
  - Number of browser history entries + top 5 domains
  - Number of files with EXIF + GPS count
  - Keyword search hits (e.g., "password: 3 hits", ".exe: 14 hits")

  "Write a professional forensic summary in Markdown with:
   1. Executive Summary (2-3 sentences)
   2. Key Findings (bullet points)
   3. Suspicious Activity (if any)
   4. Recommended Follow-up Actions
   5. Evidence Integrity Statement"
⚠️ Security note: The API key is stored as a Render.com environment secret (never in code or git). The key can also be provided per-request through the UI, stored temporarily in the browser session only.

HTML5 · CSS Custom Properties · Vanilla JS · Font Awesome · Inter Font
Why Vanilla JS (no React/Vue)? A forensic investigation tool needs reliability over developer experience. The SPA fits comfortably in a single index.html file that deploys to Cloudflare Pages in seconds with no bundler.

Core Module Reference

All forensic logic lives in backend/modules/. Each file contains a single focused class. This section documents every class, its constructor arguments, and its public methods.

modules/

analyzer.py — ForensicAnalyzer

Purpose: Detect the type of evidence file from its extension and magic bytes, compute cryptographic hashes, and return basic file metadata.

Constructor: ForensicAnalyzer(evidence_path: str) — raises FileNotFoundError if path doesn't exist.

  • detect_type()Checks extension first (e01 → E01, iso → ISO, dd/raw/img → disk_image), then falls back to magic bytes: EVF for E01, CD001 at offset 1 for ISO, 0x55AA at bytes 510–511 for MBR. Returns a string literal.
  • compute_hashes()Reads file in 64 KB chunks computing MD5, SHA-1, and SHA-256 simultaneously. Returns dict with all three digests plus file size (bytes and human-readable).
  • file_info()Returns dict with name, size, size_human, extension, and full path. Uses os.stat().
  • _human_size(n)Static helper. Converts byte count to B/KB/MB/GB/TB string.
modules/

disk_analysis.py — DiskAnalyzer

Purpose: Wrap The Sleuth Kit CLI tools to perform disk image analysis. Auto-detects whether TSK is installed and gracefully degrades.

Constructor: DiskAnalyzer(image_path: str)

  • run_mmls()Executes mmls. Parses lines into dicts: {slot, start, end, length, description, offset}. Returns {raw, partitions[], count}.
  • get_primary_offset(partitions)Filters out unallocated/extended/meta partitions and returns the start sector of the most likely primary partition.
  • run_fsstat(offset)Executes fsstat -o {offset}. Parses fs_type, volume_name, last_mount, block_size, block_count from output text.
  • run_fls(offset, deleted_only)Executes fls -r [-d] [-o offset]. Calls _parse_fls_line() for each line. Returns list of {type, inode, name, deleted, raw} dicts.
  • run_ils(offset)Executes ils. Parses pipe-delimited output into {inode, status, links, raw} dicts.
  • recover_files(output_dir, offset)Executes tsk_recover [-o offset] image output_dir. Walks output_dir afterward and returns list of recovered file paths.
  • run_full_disk_analysis()Orchestrates the entire pipeline: mmls → primary offset → fsstat → fls (all) → fls (deleted) → ils. Returns consolidated dict. Caps: 5,000 files, 2,000 deleted, 1,000 inodes.
modules/

artifact_extractor.py — ArtifactExtractor

Purpose: Walk the recovered files directory and extract browser history, EXIF metadata, multimedia catalog, and document catalog. Optionally carve media from the original evidence image.

Constructor: ArtifactExtractor(recovered_path, evidence_path=""). Auto-detects exiftool, photorec, foremost availability via shutil.which().

  • extract_browser_history()Walks recovered dir. Files named History (SQLite magic) → Chrome. Files named places.sqlite → Firefox. Edge History also detected. Returns list of {browser, url, title, visit_count, visit_time, source}. Capped at 500.
  • extract_metadata()Finds all image files (.jpg/.png/.gif/.bmp/.tiff/.webp), calls exiftool -json on each. Falls back to os.stat() timestamps. Returns list of metadata dicts. Capped at 200.
  • extract_multimedia()Catalogs all image and video files: {type, filename, path, relative_path, size, extension}. Capped at 300.
  • extract_documents()Catalogs PDF, Word, Excel, PowerPoint, TXT, CSV, RTF files with doc_type classification. Capped at 300.
  • run_media_carving(output_dir)Tries photorec first, then foremost. Returns {status, tool, carved_count, output_dir}.
  • extract_all()Calls all methods above and returns consolidated dict {browser_history, metadata, multimedia, documents, carving}.
modules/

timeline_builder.py — TimelineBuilder

Purpose: Build a chronological forensic MAC-time (Modified/Accessed/Changed) timeline of file activity. Uses TSK tools when available, filesystem stats as fallback.

Constructor: TimelineBuilder(image_path, recovered_path). Checks for fls and mactime availability.

  • build_tsk_timeline(offset)Runs fls -r -m / -o offset image to produce body file. If mactime is present, runs mactime -b body -d and parses CSV. Otherwise parses the body file directly. Returns up to 2,000 events.
  • _parse_mactime_output(output)Parses mactime CSV: date, size, activity (M/A/C/B), permissions, inode, filename. Source: "mactime".
  • _parse_body_file(content)Parses fls body format directly: MD5|name|inode|mode|UID|GID|size|atime|mtime|ctime|crtime. Converts Unix timestamps via _ts_to_date(). Source: "fls-body".
  • build_filesystem_timeline()Fallback: walks recovered directory, uses os.stat() for mtime/atime/ctime. Source: "filesystem".
  • build_timeline(offset)Master method. Uses TSK if tools exist and image is a real file, otherwise falls back to filesystem timeline.
modules/

keyword_search.py — KeywordSearchEngine

Purpose: Multi-source keyword search spanning the entire recovered evidence corpus. Each keyword can target different data sources based on its pattern.

Constructor: KeywordSearchEngine(recovered_path, artifacts) where artifacts is the output of ArtifactExtractor.extract_all().

  • search_filenames(keyword)Walks recovered dir, matches keyword (case-insensitive) against each filename. Returns up to 100 hits with {match_type:"filename", file, context}.
  • search_file_content(keyword)Opens all text-format files (≤10 MB), reads line by line, returns matching lines with line number and 200-char context. Up to 200 hits.
  • search_browser_history(keyword)Searches URL and title fields in the artifacts browser_history list. Returns {match_type:"browser_history", browser, url, title, visit_time}.
  • search_metadata(keyword)Searches all key-value pairs in the EXIF raw dict of each metadata entry. Returns {match_type:"metadata", file, context:"key: value"}.
  • search_sqlite_databases(keyword)Opens every SQLite file found in recovered dir, reads ≤20 tables × 1,000 rows, checks every string cell. Up to 50 hits.
  • search_emails(keyword)Regex email detection (\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b) across all text files. Up to 100 unique emails.
  • search_urls(keyword)Regex URL extraction (https?://... or www....) from text files. Up to 200 unique URLs.
  • search_all(keywords[])Dispatches to the right search methods based on keyword pattern: email address → search_emails; starts with . → filenames only; URL → search_urls; general → all sources. Deduplicates results.
  • search_single(keyword)Convenience wrapper around search_all for a single keyword. Returns list directly.

Smart Keyword Routing

if "@" in kw and "." in kw:     → search_emails()      # e.g., user@gmail.com
elif kw.startswith("."):         → search_filenames()   # e.g., .exe, .docx
elif kw.startswith("http"):      → search_urls()        # e.g., https://...
else:                            → all sources          # e.g., "password", "bitcoin"

Forensic Analysis Pipeline

When POST /api/analyze/{id} is called, a FastAPI BackgroundTask executes the following steps sequentially. Each step writes progress to MongoDB for live UI updates.

1

Evidence Type Detection & Hashing (0–10%)

ForensicAnalyzer.detect_type() reads magic bytes to classify: e01, iso, disk_image, or logical_file. Then compute_hashes() streams the file in 64 KB chunks computing MD5 + SHA-1 + SHA-256 simultaneously. Results saved to the evidence MongoDB document.

2

Disk Image Analysis — mmls + fsstat (10–30%)

For disk images: DiskAnalyzer.run_mmls() maps the partition table. get_primary_offset() identifies the boot partition's sector offset. run_fsstat(-o offset) extracts filesystem type, volume name, and block info. Skipped for logical files.

3

File Listing — fls (30–45%)

run_fls(offset, deleted_only=False) recursively lists all files (capped at 5,000). run_fls(offset, deleted_only=True) extracts only deleted entries (capped at 2,000) using the -d flag. run_ils() maps inode table (capped at 1,000).

4

File Recovery — tsk_recover (45–60%)

DiskAnalyzer.recover_files(output_dir, offset) runs tsk_recover -o offset image output_dir. Recovered files land in storage/recovered/{job_id}/. The directory is then walked to collect a list of all recovered file paths.

5

Artifact Extraction (60–75%)

ArtifactExtractor.extract_all() runs concurrently: (a) browser history from Chrome/Firefox SQLite files; (b) EXIF metadata from images via ExifTool; (c) multimedia file catalog; (d) document catalog. Optionally runs PhotoRec/Foremost media carving.

6

Timeline Construction (75–82%)

TimelineBuilder.build_timeline(offset): if TSK is available, generates body file with fls -m / then converts with mactime -d. Parses CSV output into date-sorted events. Falls back to filesystem stat() if TSK unavailable. Capped at 2,000 events.

7

Keyword Search (82–90%)

KeywordSearchEngine.search_all(keywords) scans filenames, text file content, browser history, EXIF metadata, and SQLite databases for each keyword provided. Smart routing dispatches email patterns, file extensions, URLs, and general terms to the appropriate search method.

8

AI Summary Generation (90–95%)

AISummarizer.generate_summary(analysis_data) constructs a structured prompt from all findings and sends to GPT-4o-mini. Returns a 5-section Markdown narrative. Falls back to template if no API key. The summary is embedded at the top of the HTML report.

9

Interactive HTML Report Generation (95–100%)

ReportGenerator.generate() assembles all results into a self-contained HTML file with embedded CSS, JavaScript, and Base64 images. Report is saved to storage/reports/{job_id}/report.html and the URL is returned as /storage/reports/{job_id}/report.html.

Security & Forensic Integrity

9.1 Evidence Integrity

Every uploaded file is hashed before analysis begins. The hashes are stored in MongoDB and displayed prominently in the report. This follows the core forensic principle: evidence must be verifiable and unaltered.

Hash AlgorithmWhy Computed
MD5Legacy compatibility — required by many law enforcement evidence forms
SHA-1Still accepted by many forensic tools and court systems
SHA-256Current gold standard — collision-resistant, recommended for new cases

Hashing uses 64 KB streaming chunks to handle disk images up to hundreds of gigabytes without loading the entire file into RAM.

9.2 Write-Blocking (Design Note)

The platform reads evidence files using Python's standard open(file, "rb") — read-only mode. Sleuth Kit's fls, mmls, fsstat tools also open disk images read-only. However, for legally admissible forensics, use a hardware write-blocker between the storage medium and the analysis workstation before creating the disk image.

9.3 API Key Security

  • OPENAI_API_KEY and MONGODB_URI stored as Render.com environment secrets — never in git
  • .env file in .gitignore to prevent accidental commits
  • API key submitted via UI is never stored server-side — only used for the duration of one analysis job
  • Cloudflare Worker shields the Render backend URL from direct exposure

9.4 Input Validation

FastAPI + Pydantic validates all request parameters. File uploads are restricted by MAX_UPLOAD_MB (default 4 GB). Analysis caps (MAX_FILES_LIST = 10,000, etc.) prevent memory exhaustion from adversarially large disk images. All subprocess commands use list-form arguments (no shell=True) preventing command injection.

9.5 CORS Configuration

Currently set to allow_origins=["*"] for development flexibility. For production, replace with your specific Cloudflare Pages domain: allow_origins=["https://forensic-platform.pages.dev"].
